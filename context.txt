## What We Built

An end-to-end pipeline: **Prompt -> LLM -> Validated Mission JSON -> Deterministic Executor -> Simulator**

The key architectural guarantee: **The LLM proposes, the validator polices, the executor is deterministic and never imports the LLM.** You can kill the LLM process and the executor still flies any valid mission from a file.

## Architecture Decisions

### Drone (not ground robot)
Drones keep all 3 challenges viable. Formation flight has more 'wow', aerial vision-follow is stronger. Closest open-source repo (monemati/PX4-ROS2-Gazebo-YOLOv8) already does drone + ROS 2 + Gazebo + YOLO.

### ROS 2 Humble (not Jazzy)
Most task-listed repos target Humble. Jazzy ecosystem (Ubuntu 24.04) hasn't fully caught up.

### Pydantic v2
Schema + custom validators in one lib. Covers Layer 1 (structural parse) and Layer 2 (semantic safety).

### MAVSDK-Python (not raw MAVLink)
Clean async API for arm/takeoff/goto/land and offboard velocity. Deterministic and auditable.

### Ollama local as default
No API key needed. Demo runs offline. Reinforces "LLM is optional" architecture story.

---

## 3-Layer Validation

**Layer 1 (Structural):** JSON parse, root object check, `actions` field exists, non-empty list
**Layer 2 (Semantic/Safety):** altitude [1-50m], speed [0.5-15 m/s], must-start-TAKEOFF,
  must-end-LAND/RTL, geofence compliance, jump distance <=500m, max 50 actions
**Layer 3 (Re-prompt):** Errors fed back to LLM for self-correction, up to 3 retries

---

## Executor State Machine

```
IDLE -> ARMING -> TAKEOFF -> EXECUTING(action_i) -> ... -> LANDING -> DONE
                              |
                              v
                           FAILSAFE
```

Properties:
- File-based: `--mission validated.json`, no LLM dependency
- Every command logged with ISO timestamp (audit trail)
- Deterministic: same JSON -> identical audit log every time
- 5s connection timeout, falls back to simulation mode if no drone

## Docker Verification Run (ultralytics/torch install in container)

Attempted `docker run` with entrypoint — **ultralytics install timed out** (torch is ~800MB, needs 20+ min).
- Entrypoint auto-installs ultralytics on first run, but default 10-min timeout was insufficient.
- **Fix**: Run container interactively first to let torch install, then run verify.sh separately.

### Entrypoint Logging Fix

Original `entrypoint.sh` used `pip install ... 2>&1 | tail -5` which hid pip download progress — only showed the last 5 lines after completion.
**Fix**: Removed `tail -5` so full install logs stream to stdout.
**Result**: Rebuilt Docker image (Step 10 invalidated cache, got new layer). Next `docker run` showed live torch download progress (532 MB, visible progress bars).
**Lesson**: Always pipe pip install directly to stdout when debugging — hidden progress wastes time.

### sympy distutils Conflict & Resolution

Base image has `sympy 1.9` from apt (distutils). pip's newer sympy 1.14.0 can't uninstall the distutils version.
**Attempted fix**: Added `--ignore-installed` flag to entrypoint.sh pip install.
**Problem**: Runtime install still failed because `--ignore-installed` wasn't surviving into the Docker image (COPY cache issue).
**Final fix**: Moved ultralytics + torch install **into Dockerfile** (baked at build time):
- Added `RUN pip install --no-cache-dir --ignore-installed sympy==1.14.0` (resolves apt conflict)
- Added `RUN pip install --no-cache-dir --ignore-installed ultralytics==8.3.0` (bakes into image)
- Removed runtime dependency from entrypoint.sh (now just a fallback + version check)
- **Effect**: Build takes ~20 min once; container starts instantly after that.
