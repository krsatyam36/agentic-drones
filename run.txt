WHY Gazebo builds FAILED
===============================================================================

The build script (`docker/setup_px4.sh`) runs `make px4_sitl gazebo-classic` and
`make px4_sitl gz_x500` with `2>/dev/null` — errors are HIDDEN. Both fail:

1. `gazebo-classic`: Not a valid make target in PX4 v1.15.4. No `gazebo` or
   `libgazebo-dev` packages are installed in the container.

2. `gz_x500`: Not a valid make target in PX4 v1.15.4. The `Tools/simulation/gz/`
   directory exists but `gz` binary is not installed (missing gz-garden package).

3. Fallback: `make px4_sitl` builds a HEADLESS PX4 binary — but it DOES include
   the `simulator_mavlink` module (confirmed: 14 references in binary). This
   module connects PX4 to any external MAVLink-based simulator.

THE FIX: jMAVSim (already built from Java source) runs externally as the
simulator. PX4's `simulator_mavlink` module connects to it over TCP:4560.
No rebuild needed — the existing `px4` binary works as-is.


HOW IT WORKS
===============================================================================

jMAVSim (TCP server :4560)  →→→  PX4 SITL (TCP client)  →→→  MAVSDK (UDP:14540)
     (headless sim)                (simulator_mavlink)           (our executor)

When PX4 starts with `PX4_SIM_MODEL=jmavsim_iris`, the startup script (rcS →
px4-rc.simulator, jMAVSim branch) runs:
    simulator_mavlink start -h localhost 4560

This connects PX4 as a TCP client to jMAVSim's TCP server on port 4560.
The iris quadrotor then performs physics simulation in jMAVSim.


Prerequisites (verify before starting):
  - Container running: docker exec -it agentic-drones-test bash
  - PX4 SITL binary: /workspace/src/PX4-Autopilot/build/px4_sitl_default/bin/px4
  - jMAVSim jar: /workspace/src/PX4-Autopilot/Tools/simulation/jmavsim/jMAVSim/out/production/jmavsim_run.jar
  - Python deps: mavsdk, pydantic, ultralytics, openai
  - OBS Studio open, configured to capture jMAVSim window + terminals

If jMAVSim jar needs building:
    cd /workspace/src/PX4-Autopilot/Tools/simulation/jmavsim/jMAVSim
    ant create_run_jar copy_res
    ls out/production/jmavsim_run.jar   # confirm ~2.3 MB jar exists


RESET STALE STATE (run once before starting):
    killall -9 px4 2>/dev/null; pkill -9 -f jmavsim 2>/dev/null
    rm -f /workspace/src/PX4-Autopilot/build/px4_sitl_default/dataman
    ipcrm -a 2>/dev/null; sleep 2; echo "Cleaned"

IMPORTANT: The px4-rc.simulator script in the build directory has been patched
to skip the internal jMAVSim launch/kill logic (since jMAVSim is run externally).
The backup is at px4-rc.simulator.backup.


================================================================================
 PANE 1 — jMAVSim (Headless Simulator)
================================================================================
jMAVSim runs in TCP server mode, accepting PX4's TCP client connection.
Use `-no-gui` for headless operation. Add `DISPLAY=:1` to get the 3D GUI.

Commands:
    cd /workspace/src/PX4-Autopilot/Tools/simulation/jmavsim/jMAVSim
    java -jar out/production/jmavsim_run.jar -no-gui -tcp 127.0.0.1:4560

Expected output:
    Options parsed, starting Sim.
    GUI not enabled
    Init MAVLink
    (TCP server listening on port 4560, waits for PX4 client)

To show 3D GUI (for OBS/demo):
    DISPLAY=:1 LIBGL_ALWAYS_SOFTWARE=1 java -jar out/production/jmavsim_run.jar -tcp 127.0.0.1:4560

Leave this running. PX4 will connect as a TCP client on port 4560.

================================================================================
 TROUBLESHOOTING
================================================================================

Problem: `make px4_sitl gazebo-classic` or `make px4_sitl gz_x500` fails
  Cause: No Gazebo/Gazebo-Classic packages installed. jMAVSim is the correct
         visualizer for this setup. See "WHY `make px4_sitl gz_x500` FAILED"
         above. Use the jMAVSim + binary approach instead of make targets.

Problem: PX4 SITL fails with "no airframe selected"
  Fix: Ensure PX4_SIM_MODEL=iris is exported before starting px4

Problem: jMAVSim shows "Could not open display" or GL errors
  Fix: Use -no-gui flag for headless operation (no X11 needed).
       For GUI: Ensure DISPLAY is set, X11 socket mounted.
       Check: echo $DISPLAY; ls -la /tmp/.X11-unix/

Problem: PX4 reports "server already running for instance 0"
  Fix: A stale PX4 instance is still holding shared memory. Run
    killall -9 px4; rm -f dataman; ipcrm -a

Problem: PX4 kills external jMAVSim on startup
  Fix: The rc.simulator was patched to skip internal jMAVSim launch.
       If re-patched, restore from: cp px4-rc.simulator.backup px4-rc.simulator

Problem: "MAVSDK connection timeout" or grpc TypeError
  Fix: Ensure MAVSDK >= 3.15.0 is installed (grpcio compatibility fix):
    pip install --upgrade mavsdk
  Verify PX4 is running BEFORE starting the executor.
  Port 14540 should receive UDP packets: timeout 3 python3 -c "
    import socket; s=socket.socket(socket.AF_INET,socket.SOCK_DGRAM)
    s.bind(('0.0.0.0',14540)); s.settimeout(2)
    data,addr=s.recvfrom(65535); print(f'{len(data)}B from {addr}')"

Problem: Executor falls back to "simulation mode"
  Fix: Check the connection string. Must use the MAVSDK 3.x API:
    python3 -m src.executor.executor --mission test/mission.json --connection "udp://:14540"
  The `system_address` parameter is now passed to `connect()` instead of
  being incorrectly passed as `mavsdk_server_address` to the System constructor.

Problem: jMAVSim jar not found or build fails
  Fix: Run ant from the jMAVSim directory as shown in prerequisites above.

Problem: LLM returns invalid mission JSON
  Fix: The 3-layer validator will catch it and re-prompt automatically.
       If retries are exhausted, try a simpler prompt.
